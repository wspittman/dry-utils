# Plan: MockAzureContainer Improvements

## Goal

Improve `MockAzureContainer` to natively handle all SQL patterns generated by the `Container` class and its `Query` builder, eliminating the need for custom matchers in common cases.

## Background

`MockAzureContainer` is the in-memory test double for CosmosDB. It supports basic patterns but requires callers to provide `MockQueryDef` entries (with `matcher` + `func`) for anything beyond trivial queries. Since `Query.build()` generates predictable, structured SQL, the mock can be smarter about evaluating it natively.

## Current Native Support

| Pattern                                                                                        | Status                                 |
| ---------------------------------------------------------------------------------------------- | -------------------------------------- |
| `SELECT * FROM c`                                                                              | Ôťů Handled                             |
| `SELECT VALUE COUNT(1) FROM c`                                                                 | Ôťů Handled                             |
| `SELECT c.x, c.y FROM c` (simple projection, no alias)                                         | Ôťů Handled                             |
| `SELECT * FROM c WHERE (...)`                                                                  | ÔŁî Falls through to custom matchers    |
| `SELECT VALUE COUNT(1) FROM c WHERE (...)`                                                     | ÔŁî Falls through                       |
| `SELECT TOP n * FROM c WHERE (...)`                                                            | ÔŁî Falls through                       |
| `SELECT c.x AS name, COUNT(1) AS count FROM c WHERE IS_DEFINED(c.x) GROUP BY c.x` (getCountBy) | ÔŁî Falls through (returns wrong shape) |

## SQL Patterns Produced by Container/Query

All patterns produced by `Query.build()`:

- `SELECT [TOP n] * FROM c [WHERE (clause1) AND (clause2) ...]`
- `SELECT [TOP n] c.id FROM c [WHERE ...]`
- `SELECT [TOP n] VALUE COUNT(1) FROM c [WHERE ...]`

WHERE clauses from `Query.whereCondition`:

- `c.field = @field` (and `<`, `<=`, `>`, `>=`)
- `CONTAINS(c.field, @field, true)`
- Multiple clauses AND-joined with parentheses: `WHERE (c.x > @x) AND (c.y = @y)`

Raw SQL from `Container`:

- `getCountBy`: `SELECT c.{prop} AS name, COUNT(1) AS count FROM c WHERE IS_DEFINED(c.{prop}) GROUP BY c.{prop}`

## Phases

### Phase 1: Implement native WHERE evaluation [ ]

**Scope:** `mockAzureContainer.ts` ÔÇö `_query` method.

Implement a `evaluateWhere` helper that:

1. Parses WHERE clause string from `SELECT ... FROM c WHERE (...)` format
2. Extracts individual conditions from AND-joined parenthesized groups
3. Evaluates each condition against a `SqlQuerySpec`'s `parameters` array
4. Supports operators: `=`, `<`, `<=`, `>`, `>=`
5. Supports `CONTAINS(c.field, @param, true)`

Update `_query` to:

- Detect and handle `SELECT * FROM c WHERE (...)` Ôćĺ filter items + return all properties
- Detect and handle `SELECT VALUE COUNT(1) FROM c WHERE (...)` Ôćĺ filter + return count
- Detect and handle `SELECT TOP n * FROM c [WHERE ...]` Ôćĺ filter + slice

### Phase 2: Handle `getCountBy` natively [ ]

**Scope:** `mockAzureContainer.ts` ÔÇö `_query` method.

Add a regex for: `SELECT c.{prop} AS name, COUNT(1) AS count FROM c WHERE IS_DEFINED(c.{prop}) GROUP BY c.{prop}`

Group items by the field value and return `{ name, count }` pairs.

### Phase 3: Update tests [ ]

**Scope:** `container.test.ts`

- Remove custom `matcher` entries from `getContainer()` that are now handled natively
- Add test cases for:
  - `query` with a `Query` instance that has a WHERE condition
  - `getCountBy`
  - `query` with `TOP` (via `Query.top()`)

## Key Decisions

- **No SQL parser library** ÔÇö all patterns are predictable and regex-parseable
- **Keep `MockQueryDef` as escape hatch** ÔÇö custom matchers remain for non-standard queries
- **Only implement what `Container` + `Query` actually produce** ÔÇö no over-engineering
